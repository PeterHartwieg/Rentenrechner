# Crawler policy — RentenWiki.de

How RentenWiki.de signals to search and AI crawlers what they may do with the
public site. The policy is enforced in two layers (Cloudflare CDN + origin
`robots.txt`) and is intent-signalling by design — it is not a hard
enforcement mechanism for misbehaving bots.

> Issue: `.scratch/seo-geo-discovery/issues/09-cloudflare-managed-robots-content-signals.md`
> Lock-in source: `.scratch/seo-geo-discovery/issues/01-lock-seo-scope-architecture-decisions.md`
> Last verified live: 2026-05-06

## 1. Locked policy

```
search=yes
ai-input=yes
ai-train=no
```

Meaning:

- **search=yes** — search-engine crawlers (Googlebot, Bingbot, Applebot) may
  fetch and index the site.
- **ai-input=yes** — AI assistants may fetch a page at request time to quote
  or cite it in an answer (OAI-SearchBot, ChatGPT-User, PerplexityBot).
- **ai-train=no** — training crawlers (GPTBot, ClaudeBot, anthropic-ai,
  Google-Extended, CCBot) are not consented to and must not use site content
  to train models.

**Why this shape.** The project is a public, free tool published under
PolyForm Noncommercial 1.0.0 (`LICENSE.md`). The license posture is "publish
openly for humans and search/answer crawlers; do not consent to model-training
scraping." `ai-train=no` is the codified signal of that posture; `search=yes`
+ `ai-input=yes` keep the site discoverable through both classical search and
modern answer engines.

This policy is locked. Changing any of the three flags is a license-posture
decision, not a routine config change — re-open issue #01 first.

## 2. Implementation layers

The policy is enforced in two layers that compose. The served
`https://rentenwiki.de/robots.txt` is Cloudflare's managed block prepended to
the project's origin `robots.txt`; both layers are visible in a single
response.

### Layer 1 — Cloudflare zone (CDN)

Cloudflare's zone-level Bot Management settings:

- **`ai_bots_protection: "block"`** — Cloudflare's signature-based detection
  drops requests from known AI training bots (GPTBot, ClaudeBot, CCBot,
  Bytespider, …) at the edge before they reach origin. This is what gives the
  policy any teeth against bots that ignore robots.txt — Cloudflare matches
  on UA + IP heuristics, not on the bot's promise to honour `Disallow`.
- **`is_robots_txt_managed: true`** — Cloudflare prepends a managed Content
  Signals preamble + Disallow block to whatever the origin returns at
  `/robots.txt`. This is the layer that emits the literal
  `Content-Signal: search=yes,ai-train=no` header line and the curated
  Disallow list (Amazonbot, Applebot-Extended, Bytespider, CCBot, ClaudeBot,
  CloudflareBrowserRenderingCrawler, Google-Extended, GPTBot,
  meta-externalagent, …). Cloudflare maintains the bot list independently —
  we get updates without code changes.

### Layer 2 — Origin `dist/robots.txt`

Generated by `src/seo/robots.ts` from two constants and the
`ORIGIN_CONTENT_SIGNAL` triple:

- `ALLOWED_BOTS` → `Googlebot`, `Bingbot`, `Applebot`, `OAI-SearchBot`,
  `ChatGPT-User`, `PerplexityBot` — each gets an explicit `Allow: /` block so
  hosts that default-deny unknown bots still let our allow-list through.
- `DISALLOWED_BOTS` → `GPTBot`, `ClaudeBot`, `anthropic-ai`,
  `Google-Extended`, `CCBot` — each gets an explicit `Disallow: /` block, so
  the intent is in the file even if Cloudflare's managed block is ever
  toggled off.
- `ORIGIN_CONTENT_SIGNAL` = `search=yes,ai-input=yes,ai-train=no` — emitted
  in the wildcard `User-agent: *` block so signal-aware parsers see the full
  documented triple even when Cloudflare's preamble emits only
  `search=yes,ai-train=no`. Per RFC 9309 §2.2.1, duplicate `User-agent: *`
  records are merged into one group, so both `Content-Signal:` lines apply.
  We emit our triple here because Cloudflare's variant cannot be flipped
  via API and the dashboard-exposed variants only differ in Disallow
  phrasing (not Content-Signal flags).

The file ends with `User-agent: * / Content-Signal: … / Allow: /` so
unrecognised but legitimate crawlers (Yandex, DuckDuckBot, future search
engines) are not collateral and so signal-aware bots see the documented
triple.

Tests in `src/seo/robots.test.ts` pin the byte-identical output. When the
allowed/disallowed lists or the Content-Signal triple change, the test
snapshot updates with them; the policy stays grep-able from
`src/seo/robots.ts` alone.

### Signal split — who emits what

| Signal       | Cloudflare preamble | Origin (this repo) |
| ------------ | ------------------- | ------------------ |
| `search=yes` | yes                 | yes                |
| `ai-input=yes` | (silent)          | yes                |
| `ai-train=no` | yes                | yes                |

Cloudflare's preamble does not currently emit `ai-input` (default variant,
2026-05). Our origin block fills the gap so the served `/robots.txt`
contains the full locked triple in at least one `User-agent: *` group.

## 4. Live verification

After any change to `src/seo/robots.ts`, the Cloudflare dashboard, or as a
periodic audit, run these checks against the live domain.

### Visual inspection

```bash
curl -s https://rentenwiki.de/robots.txt | head -50
```

Expect to see Cloudflare's managed preamble (a `Content-Signal:` line and a
generous Disallow block for known training bots) **followed by** the
project's `Sitemap:` line and per-bot Allow/Disallow blocks.

### Scripted verification (no credentials needed)

```bash
node scripts/apply-cloudflare-policy.mjs verify
```

Probes the live domain with allowed and disallowed bot user-agents and
asserts that each disallowed bot has a corresponding `Disallow: /` line in
the served `/robots.txt`. Fails non-zero on regression.

### Scripted status read (requires credentials)

```bash
CLOUDFLARE_API_TOKEN=… CLOUDFLARE_ZONE_ID=287d66fbc1bbed67034cf0dfa92dc812 \
  node scripts/apply-cloudflare-policy.mjs status
```

Reads the zone's current `bot_management` settings and prints the JSON. Use
this to confirm the values in the table above haven't drifted.

## 5. Maintenance

When something needs to change, here's where it lives:

- **Add or remove a bot from the allow/disallow list** — edit `ALLOWED_BOTS`
  or `DISALLOWED_BOTS` in `src/seo/robots.ts`, update tests, ship. Cloudflare's
  managed Disallow list is updated independently (Cloudflare-controlled); the
  two compose, so adding a bot only at the origin layer is fine.

- **Want to flip `crawler_protection` on** — by default the script's
  `POLICY_BODY` only re-affirms `ai_bots_protection: block` (so re-running
  `apply` is a no-op against the live-verified state). To enable
  `crawler_protection`, edit `POLICY_BODY` in `scripts/apply-cloudflare-policy.mjs`
  to add `crawler_protection: 'enabled'`, then run
  `node scripts/apply-cloudflare-policy.mjs apply`. Be aware this can fire
  managed challenges (CAPTCHA, JS) at legit users on shared, corporate-NAT,
  or VPN IPs. Watch the analytics for a few days after enabling. To revert,
  remove the field from `POLICY_BODY` and re-run, or flip via the dashboard.

- **Change `cf_robots_variant`** — dashboard only. Security → Bots → Crawl
  Control / Managed robots.txt. The variants (e.g. `off`, more permissive
  copies) are not exposed via the script's `apply` body.

- **Renew the API token** — Cloudflare profile → API Tokens → create a token
  with `Zone:Read` + `Zone:Bot Management:Edit` scopes restricted to the
  rentenwiki.de zone. Store it in operator credentials, not in the repo.

- **Free-plan limitation** — if `apply` or `status` returns HTTP 403/400
  with a plan-related error, the AI-bot-protection subset has been gated
  behind a paid tier. Note the date here, decide whether to upgrade or fall
  back to origin-only enforcement.

## 7. Threat model and limitations

What this policy is and isn't:

- **Robots.txt is honour-based.** Bots that ignore robots.txt aren't blocked
  by Layer 2 at all. Layer 1 (Cloudflare's signature detection) is what
  catches well-known training bots that ignore the file. Anything novel,
  rotating UA strings, or coming from residential IP pools will get through
  both layers. We accept that.
- **`ai-train=no` is intent, not enforcement.** Even with both layers, a
  determined scraper can always pull the public site. The policy's value is
  legal/normative: a public, machine-readable, timestamped record of "the
  site explicitly does not consent to training use." That matters for the
  PolyForm Noncommercial license posture and for any future enforcement
  conversation.
- **No PII trade-off.** Neither layer collects user data, sends telemetry,
  or sets cookies on legitimate visitors. Cloudflare's bot detection runs on
  network-level signals (IP reputation, TLS fingerprint, UA), not on
  per-visitor profiling.
- **This is a public-publication site.** The whole content surface is
  intended to be readable by humans and search/answer crawlers without
  authentication. We don't gate articles. If a bot reaches the site, it's
  reading the same HTML a browser gets — there is no privileged content to
  protect.

If a future feature requires more aggressive bot mitigation (e.g. rate
limiting on a calculator API endpoint), that's a new policy decision —
re-open this doc and add a section.
