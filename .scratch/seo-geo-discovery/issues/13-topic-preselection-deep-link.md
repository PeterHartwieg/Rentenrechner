## What to build

A small `?topic=<slug>` mechanism that lets SEO topic pages deep-link into the calculator with the right view (compare / combine) preselected and the right products pre-visible. Bridges issues #02–#07's CTAs from "go to `/` and start over" to "land on the right comparison".

This re-introduces a scoped version of the deprecated trigger router that CLAUDE.md, BACKLOG.md, and AGENTS.md still reference (`PATH_OPTIONS` / `VISIBLE_PRODUCTS_BY_PATH` / `WIZARD_REGISTRY`) but which no longer exists in code. When this slice lands, follow up by sweeping the topic-page CTAs in #02–#07 to use `?topic=<slug>` instead of plain `/`.

## Acceptance criteria

- [ ] A typed topic→preselection table maps a slug (e.g. `rentenluecke`, `etf-vs-bav`, `riester`, `altersvorsorgedepot`, `riester-vs-altersvorsorgedepot`, `basisrente`, `private-rentenversicherung`, `rente-netto`, `altersvorsorgeprodukte-vergleichen`, `bav-rechner`) to `{ mode: 'compare' | 'combine', visibleProducts?: ProductId[] }` (and any other seeds the calculator needs to land on the right state).
- [ ] `LandingPage` reads `?topic=` on mount; if the slug matches a table entry, auto-fires `onChoice` with the right mode and the workspace seeds `visibleProducts` accordingly.
- [ ] If `?topic=` is missing or unrecognized, `LandingPage` renders normally (no error, no auto-fire).
- [ ] Returning users (saved state) are not overridden by `?topic=` — preselection only applies to the first-time landing flow, so a returning user's saved workspace is never disturbed (PRD story 18).
- [ ] Unit tests cover: known slug → correct mode + visibleProducts; unknown slug → normal landing render; returning user with saved state → saved state wins over `?topic=`.
- [ ] CLAUDE.md, BACKLOG.md, and AGENTS.md references to `PATH_OPTIONS` / `VISIBLE_PRODUCTS_BY_PATH` / `WIZARD_REGISTRY` are updated to reflect the new mechanism (or removed if they describe an older deprecated design).
- [ ] After this slice merges, a follow-up commit (this issue or a sibling sweep) replaces the topic-page CTAs from `#02`–`#07` with `?topic=<slug>` deep-links.
- [ ] `npm run verify` passes.

## Blocked by

None — can start independently of #02. Best paired with the content release wave so topic pages can use the deep-link CTAs from day one, but technically standalone.

### Agent Brief (2026-05-06)

**Category:** enhancement
**Summary:** Add a `?topic=<slug>` deep-link mechanism so SEO topic pages can route users into the calculator with the right view (compare / combine) and product preselection. The slug → preselection mapping lives as a typed field on the existing `PublicRoute` interface in `src/seo/publicRouteRegistry.ts` (one source of truth per route). On `/`, when `?topic=<slug>` matches a registered route AND the user has no saved state, fire `onChoice` with the route's preselection embedded. Also sweep `/rentenluecke-rechner`'s CTA from plain `/` to `/?topic=rentenluecke-rechner`.

**Current behavior (post-#02 merge, pre-#03):**
`LandingPage.onChoice` accepts `LandingChoice = { kind: 'compare' } | { kind: 'combine' }` with no preselection. `handleLandingChoice` in `App.tsx` calls `portfolioState.setMode(...)` then either flips to `appView === 'compare'` (compare path) or shows the InventoryWizard (combine path). The wizard exposes no `initialEnabledProducts` prop. The CTA in `RentenluckeRechnerPage` points to plain `/`.

`CLAUDE.md`, `BACKLOG.md`, `AGENTS.md` reference a deprecated trigger router (`PATH_OPTIONS` / `VISIBLE_PRODUCTS_BY_PATH` / `WIZARD_REGISTRY`) that no longer exists in code (only in `.scratch/archive/scenario-led-portfolio/issues/17-trigger-router-infrastructure.md`).

**Desired behavior:**
The `PublicRoute` interface gains an optional `preselection?: { mode: 'compare' | 'combine', visibleProducts?: ProductId[] }` field. Each topic-page registry entry declares its own preselection. On `/` mount, `LandingPage` reads `?topic=<slug>` from `window.location.search`. If the slug matches a registered route AND `detectSavedMode()` returns null AND the matched route has a `preselection`, fire `onChoice` with the embedded mode and visibleProducts. For compare mode, `handleLandingChoice` seeds the workspace's `visibleProducts`. For combine mode, the InventoryWizard opens with the listed products checked in its product checklist (existing wizard UX preserved; user still walks through personal-details + contracts). For returning users, `?topic=` is silently ignored — saved state always wins (PRD story 18 literal reading).

**Key interfaces this slice touches:**

- `PublicRoute` interface — extend with `preselection?: { mode: 'compare' | 'combine', visibleProducts?: ProductId[] }`. Existing entries (`/`, `/rentenluecke-rechner`, `/404`) get a `preselection` only where it makes sense (the homepage `/` does not need one — visiting `/?topic=foo` works because the resolver uses the slug, not the route at `/`).
- `LandingChoice` type — extend each variant with optional `visibleProducts?: ProductId[]`. Both `kind: 'compare'` and `kind: 'combine'` carry the optional preselection.
- `handleLandingChoice` in `App.tsx` — when `choice.visibleProducts` is set, seed the workspace `visibleProducts` for compare mode, or pass to the InventoryWizard for combine mode.
- `InventoryWizard` Props — add `initialEnabledProducts?: ReadonlyArray<ProductId>`. Wizard uses this to seed the checked state of product rows on first render. GRV stays universally checked (already cannot be unchecked).
- A small resolver, e.g. `resolveTopicPreselection(searchString: string): { mode, visibleProducts? } | null` — pure function over `URLSearchParams` and the registry. Lives near `publicRouteRegistry.ts` so SEO consumers can share it.
- `RentenluckeRechnerPage` CTA — replace `href="/"` with `href="/?topic=rentenluecke-rechner"`. Keep the link descriptive and accessible.
