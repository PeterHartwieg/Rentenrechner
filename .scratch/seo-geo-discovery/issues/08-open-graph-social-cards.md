## What to build

Generate route-specific Open Graph / Twitter card images for each public discovery page from registry data. Either build-time SVG → PNG via a small script (preferred — deterministic, version-controlled), or static per-page assets authored once. Each card shows the brand (RentenWiki.de) and the page-specific topic so shared links look trustworthy.

## Acceptance criteria

- [ ] Each registered public route has a route-specific OG image referenced via `og:image`.
- [ ] Cards consistently render brand + page topic; brand language is RentenWiki.de (no "Rentenrechner" in card copy).
- [ ] OG image generation is deterministic (regenerating from the same registry state produces identical bytes) so reviews can catch accidental churn.
- [ ] Twitter card metadata (`summary_large_image`) references the same image.
- [ ] Snapshot tests cover OG/Twitter metadata for all 11 initial routes.
- [ ] OG image dimensions follow recommended sizes (e.g. 1200×630).
- [ ] Sitemap is unaffected; canonical and route metadata unchanged.
- [ ] `npm run verify` passes.

**Design decisions (locked):**

| Decision | Choice |
|----------|--------|
| SVG → PNG tool | `@resvg/resvg-js` — Rust-backed, no browser, deterministic output, CI-safe. Add as devDependency only. |
| Card template | Programmatic SVG string per route, generated in the script. Layout: RentenWiki.de wordmark top-left, H1 title centered, "Stand 2026" bottom-right. Simple brand colors. No external font embedding — use system/generic sans-serif stack in SVG `font-family`. |
| No new registry field | Convention over config: head-metadata emitter computes the per-route image path as `/og/${slug}.png` where `slug = canonical.slice(1)` (e.g. `/bav-rechner` → `/og/bav-rechner.png`). Homepage canonical `/` maps to `/og/home.png`. This requires no `PublicRoute` interface change and no per-route `ogImage` field. |
| `/404` route | Exempt — reuses `OG_DEFAULT_IMAGE_PATH`; script skips it. |
| PNGs committed | Yes — `public/og/*.png` committed to git. Deterministic generation means CI diff (`git diff --stat`) catches accidental churn. |
| Build integration | New script `scripts/generate-og-images.mjs`. Add as a pre-build step: `"prebuild": "node scripts/generate-og-images.mjs"` (or inline into `package.json build` script). Also add to `npm run verify` pipeline. |
| Testing | Snapshot tests cover the rendered HTML `og:image` and `twitter:image` meta tag content for every in-sitemap route. **Do not byte-compare PNG files** — that's brittle and unnecessary given deterministic generation. |

**What the script does:**
1. Imports `publicRouteRegistry` entries (via a dynamic `import()` of the compiled output or direct TS-via-jiti execution).
2. For each in-sitemap route (except `/404`): derives `slug`, generates an SVG string using `h1` and `title` from the registry entry, converts to PNG via `@resvg/resvg-js`, writes to `public/og/${slug}.png`.
3. Also writes `public/og/home.png` for the `/` route.
4. Logs a summary (route → file, byte size) and exits 0.

**Head-metadata change:**
In the component(s) that emit `og:image` and `twitter:image` meta tags, replace the static `OG_DEFAULT_IMAGE_PATH` reference with the computed path:
```ts
function routeOgImagePath(route: PublicRoute): string {
  const slug = route.canonical === '/' ? 'home' : route.canonical.slice(1)
  return route.inSitemap && route.canonical !== '/404'
    ? `/og/${slug}.png`
    : OG_DEFAULT_IMAGE_PATH
}
```
Keep `OG_DEFAULT_IMAGE_PATH` as the share-URL / fallback constant.
