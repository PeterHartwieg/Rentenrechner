# Temporary footer button to activate QA mode for non-technical testers

Status: done
Type: enhancement
Priority: minor

## Parent

.scratch/qa-feedback-mode/PRD.md

## Problem

QA mode can be activated three ways today:

1. URL parameter `?qa=1` (original).
2. Keyboard shortcut `Ctrl+Shift+.` / `тМШ+Shift+.` (issue 15, shipped).
3. The chip itself, but only to *deactivate* тАФ there is no visible activator anywhere in the UI when QA mode is off.

Both existing entry points assume the tester is technical: friends-and-family / less-technical QA testers do not edit URLs, do not know keyboard shortcuts exist, and miss the on-page hint entirely (the indicator chip is hidden until QA mode is already on, so the shortcut hint sits inside something invisible). This blocks the goal of "anyone can give targeted feedback during the soft-launch QA period".

We want a discoverable, click-to-activate entry point that disappears as soon as the public-facing rollout is over тАФ i.e. a *temporary* control that is easy to remove later without touching the QA mode infrastructure itself.

## Approach

Add a small "QA-Modus starten" button to `LegalFooter` (`src/features/legal/LegalFooter.tsx`), rendered next to the Lizenz hint. Footer is the lowest-stakes place on the page тАФ it doesn't compete with the calculator UI, and it's already visible on every public route through `App.tsx`.

The button is gated by a feature flag so it can be flipped off without a code change once the QA window closes. Two options:

- **Build-time flag** (preferred): a `VITE_QA_FOOTER_BUTTON` env var checked at module load. Default off in production unless explicitly enabled. Flipping it off is a one-line `.env` change + redeploy.
- **Runtime flag**: read from a `localStorage` key. Easier to flip per browser but visible to all users тАФ rejected because we want this on for everyone during the QA period.

Use the build-time flag. Document in `LegalFooter.tsx` that this is a temporary surface attached to a flag, with a TODO referencing this issue so future-us removes it cleanly.

## What to change

1. **Feature flag** тАФ Read `import.meta.env.VITE_QA_FOOTER_BUTTON === 'true'` (or the convention already in use тАФ check if any other env-flagged feature exists) at the top of `LegalFooter.tsx`. Default off if unset.

2. **Button** тАФ When the flag is on and `useQaMode().enabled === false`, render a small button styled like the Impressum/Datenschutz links: `QA-Modus starten`. On click, call `activate()` from `useQaMode`. Hide the button when QA mode is already on (the indicator chip serves as the "active" UI in that state).

3. **Copy** тАФ German-only to match the rest of the calculator UI: `QA-Modus starten`. Tooltip: `Aktiviert den Feedback-Modus. Klicken Sie anschlie├Яend ein UI-Element an, um Feedback zu geben.`

4. **Discoverability text** тАФ Optional: a short helper line above or beside the button explaining what QA mode is, so non-technical testers understand what they're activating. E.g. `"Sie testen f├╝r uns? QA-Modus starten тАФ dann auf das fragliche Element klicken."` Keep it brief; the footer is not the place for a manual.

5. **`useFeedbackTarget`** тАФ Yes, instrument the new button itself (id `legal.footer.qa-activate`) so it appears in feedback bundles when one is captured before activation (edge case but consistent with other footer items).

6. **Removal plan** тАФ In `LegalFooter.tsx`, add a TODO comment with this issue's path noting that the button + flag should be deleted once the public QA window closes. Keep the diff small enough that removal is an obvious revert.

7. **Tests** тАФ Add a `LegalFooter.test.tsx` (or extend the existing one if any) that:
   - Renders with the flag off тЖТ button absent.
   - Renders with the flag on, QA off тЖТ button present, click activates QA mode.
   - Renders with the flag on, QA on тЖТ button hidden.
