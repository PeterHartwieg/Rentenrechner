# Popups, popovers, and modals must be QA-selectable

Status: done
Type: bug
Priority: major

## Parent

.scratch/qa-feedback-mode/PRD.md

## Problem

Found in real QA testing (2026-05-06): when a popup, popover, or modal is open, QA mode cannot pin anything inside it. Testers report a layout/copy issue *inside* the wizard or the audit modal and have no way to attach the feedback to the actual element.

Concrete cases:

- **`InfoTip` popover (`src/ui/InfoTip.tsx`)** — the trigger button carries `data-qa-target` (good), but the popover content rendered when the user opens it does not. The tooltip body is invisible to QA mode, so testers cannot select the explanation text or its formatting.
- **`OptimiereVorsorgeModal` (`src/features/dashboard/`)** — the audit modal that wraps per-contract decisions has zero `useFeedbackTarget` calls (`grep -rl useFeedbackTarget src/features/dashboard/` returns nothing). All step-machine controls inside the modal are unreachable.
- **`InventoryWizard` (`src/features/inventory/`)** — same situation: zero instrumentation. Every step (Vintage chips, per-product cards, finishing screen) is invisible to QA mode.
- **`MonteCarloPanel` distribution popover** — the panel itself is instrumented, but interactive sub-controls inside are not consistently selectable.

Why this is more than "missing instrumentation": modals, wizards, and popovers are exactly where new copy and complex interactions live, so they are *also* exactly where testers want to leave feedback most often. A QA mode that goes blind whenever a dialog opens is worse than no QA mode for these surfaces — testers will silently switch to email, defeating the local-issue export feature shipped in issue 14.

There is also a possible overlay/z-index concern. `QaOverlay` uses `position: fixed` outlines, but if a modal renders above the overlay (higher z-index), the outline could be hidden under the modal. Needs to be verified during this fix and documented in `QaOverlay.tsx` so future modals don't regress it.

## What to change

1. **Popover content targets** — In `InfoTip.tsx`, add a separate `useFeedbackTarget` for the popover body when it is open (id `<feedbackTargetId>.popover`). The trigger keeps its existing target. Keyboard users who Tab into the popover content should also see the outline.

2. **Modal instrumentation** — Add `useFeedbackTarget` calls to:
   - `OptimiereVorsorgeModal` (modal container, step heading, primary CTA, each per-contract decision row).
   - `InventoryWizard` (each step container, plus leaf targets for each card / chip / button — coordinate with issue 16's per-product coverage).
   - Any other modal/dialog that renders via `role="dialog"` (search `aria-haspopup` / `role="dialog"` to enumerate).

3. **Overlay z-index audit** — Verify `QaOverlay`'s outline draws above the topmost modal layer. If a modal currently sits above the overlay, raise the overlay's z-index (or render via a portal at the document body) so the outline is always visible. Document the chosen layering rule in `QaOverlay.tsx` and add a test that mounts a modal + overlay together and asserts the outline is on top.

4. **Composer interaction with open popups** — When a popover or modal is open and the tester pins a target inside it, the composer should not auto-close the popover/modal (otherwise the tester cannot iterate). Audit `QaFeedbackProvider` `pickTarget` flow for any focus-stealing or click-outside handlers that would dismiss the host popup. Add a test for the InfoTip case (open tip → pin its content → composer opens → tip stays open).

5. **Coverage test extension** — Extend `coverage.test.tsx` (or add a sibling test) that opens each known modal/popover and asserts at least one pinnable target exists inside it.

## Acceptance criteria

- [ ] Opening an `InfoTip` and clicking inside its popover body pins the popover content as the feedback target.
- [ ] Every step inside `OptimiereVorsorgeModal` and `InventoryWizard` is pinnable at leaf level.
- [ ] Pinning a target inside a popover or modal does not close the host popup; the tester can keep iterating until they submit or cancel.
- [ ] QA overlay outline draws on top of the highest-z modal layer (regression test in place).
- [ ] `coverage.test.tsx` (or a new modal-coverage test) fails if a new modal is added without QA targets.
