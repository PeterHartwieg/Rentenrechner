# QA target coverage gaps for input fields and cards

Status: done
Type: bug
Priority: major

## Parent

.scratch/qa-feedback-mode/PRD.md

## Problem

Found while running a real QA session (2026-05-06): many text fields and cards in the live UI still cannot be pinned in QA mode. The overlay either does nothing on hover or falls back to a section-level highlight that covers a whole panel — exactly the "describe which element you mean" failure mode that issue 13 was meant to eliminate.

Issue 13 (finer-selection-granularity) is marked `done`, but only the bAV/ETF input branches and the results charts/tables actually got leaf-level `useFeedbackTarget` calls. A `grep -rl useFeedbackTarget src/features/` shows zero coverage in:

- **Per-product inputs** — `InsuranceInputs`, `BasisrenteInputs`, `RiesterInputs`, `AvdInputs`, `GrvInputs` (only `BavInputs` and `EtfInputs` are instrumented).
- **Reusable input sections** — `FeeSection`, `BeitragsdynamikField`, `OfferCapitalCompareField` (only `PayoutModeSection` is instrumented).
- **Dashboard cards** — `RecommenderCard`, `ContractDecisionCards`, `ContractDecisionMenu` (recommender card and the three decision cards weiterführen / kündigen / übertragen).
- **Result metric cards** — individual `<ResultMetric>` instances inside `ResultsPanel`, `CashFlowsPanel`, `AssumptionsPanel`. Charts have leaf coverage but the metric cards above them don't.
- **Workspace chrome** — product tabs, compare/combine mode toggle.

Section-level fallback is in place for the surrounding panels, so the overlay technically resolves *something*, which is why issue 13 looked complete from the outside. From a tester's seat it still feels broken because the highlighted box is always the panel, never the field they are pointing at.

## What to change

1. **Per-product input components** — Add `useFeedbackTarget` to every label + numeric input + select in `InsuranceInputs`, `BasisrenteInputs`, `RiesterInputs`, `AvdInputs`, `GrvInputs`, mirroring the pattern already used in `BavInputs`. Target id convention: `inputs.<product>.<field>` (see issue 01).

2. **Remaining reusable input sections** — Instrument `FeeSection`, `BeitragsdynamikField`, `OfferCapitalCompareField` the same way `PayoutModeSection` already is. These are shared across multiple per-product input panels, so one edit unlocks coverage everywhere.

3. **Dashboard cards** — Add targets to `RecommenderCard` (header, score, primary CTA), each card in `ContractDecisionCards` (weiterführen / kündigen / übertragen), and the host `ContractDecisionMenu`. Card-level target plus leaf targets for headline / amount / CTA.

4. **Result metric cards** — Add `useFeedbackTarget` to every individual `<ResultMetric>` rendered in `ResultsPanel`, `CashFlowsPanel`, `AssumptionsPanel`. The metric component itself is the right injection point so a single edit covers all callers.

5. **Workspace chrome** — Product tab labels, compare/combine mode toggle in the workspace header.

6. **Coverage test** — Extend `src/features/qa-feedback/__tests__/coverage.test.tsx` to assert that for each product id in `PRODUCT_REGISTRY`, the corresponding input panel renders at least one `data-qa-target` per field. This is the regression guardrail that would have caught the gap left by issue 13.

## Acceptance criteria

- [ ] Hovering any visible numeric input, label, dropdown, or card in QA mode pins that exact element — not the surrounding panel.
- [ ] Every product input panel (bAV, ETF, Insurance, Basisrente, Riester, AVD, GRV) has leaf-level targets for each field.
- [ ] `RecommenderCard` and the three contract-decision cards are individually pinnable.
- [ ] Each `<ResultMetric>` in the result/cashflow/assumption panels is individually pinnable.
- [ ] `coverage.test.tsx` fails when a product input panel is added without leaf-level targets.
- [ ] Section-level fallbacks remain for genuinely uninstrumented areas (e.g. legal pages already covered separately).

Nothing — additive instrumentation, no API changes.
