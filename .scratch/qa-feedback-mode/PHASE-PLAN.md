# QA feedback mode - parallel phase plan

Status: needs-triage
Created: 2026-05-05

## Goal

Sequence the QA feedback mode issues so implementation can start while a separate UI/UX QA pass is happening in parallel. The plan prioritizes work that adds new QA infrastructure with minimal edits to existing UI copy/layout files, and deliberately delays broad target instrumentation that would conflict with active QA fixes.

## Summary

Start now with the local screenshot-backed tracer bullet and its supporting privacy/accessibility/export work. Avoid broad UI instrumentation until the current QA implementation pass slows down or lands, because target coverage work touches the same components that QA fixes are likely to edit.

## Conflict watchlist

Likely low-conflict areas:

- New QA feedback module files.
- Report payload builder and redaction helpers.
- Screenshot adapter.
- Export/bundle helpers.
- Focus and composer tests.

Likely high-conflict areas during QA fixes:

- Product input components.
- Reusable field sections.
- Chart components.
- Legal/disclaimer text.
- Workspace layout/chrome.
- Any file receiving copy or visual polish from QA feedback.

## Working rule for this phase

During the active QA pass, prefer infrastructure that makes QA reports better. Delay broad instrumentation that edits many existing UI surfaces. The tracer bullet should prove the workflow with screenshots first; coverage can widen once the app surface is less volatile.
